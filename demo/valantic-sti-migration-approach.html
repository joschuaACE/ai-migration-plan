Legacy-to-Modern Migration Methodologyv3.0

Valantic STI's engineering methodology for migrating legacy applications to modern target platforms — a structured, AI-augmented, quality-gated process delivering behaviorally equivalent code with full architectural documentation. Language-agnostic. Technology-adaptive.

Executive Summary

Valantic STI has developed a repeatable, industrialized methodology for migrating legacy codebases to modern platforms. The approach combines rigorous software engineering principles with AI-augmented execution, producing production-ready code that preserves 100% of business logic while modernizing the architecture.

This methodology is language-agnostic and technology-adaptive. It applies equally to migrations such as COBOL → Java, C++ → Kotlin, Delphi → .NET, monolith → microservices, or any legacy-to-modern transformation. The principles remain constant; the technology-specific standards are configured per engagement.

Key differentiators:

  • Behavioral equivalence guarantee — same inputs produce same outputs, verified through multi-dimensional analysis
  • Architecture-first migration — code lands in a proven target architecture from day one, not refactored later
  • 10 automated quality gates per module — compilation, tests, architecture rules, coverage, semantic equivalence, drift detection
  • Knowledge graph-driven understanding — source code is analyzed and understood before any translation begins
  • Full traceability — every decision recorded, every file tracked, complete audit trail for compliance
  • Client ecosystem adaptation — target standards, naming conventions, security models, and deployment patterns adapted to each client's existing ecosystem
  • Automatic architecture documentation — complete ARC42 docs generated as a byproduct, not extra work

Scope of This Document

This document describes the Valantic STI engineering methodology for legacy-to-modern migration engagements. It covers methodology, process, architecture approach, quality assurance, tooling, risk management, and deliverables. It is technology-agnostic — specific source/target technology choices are configured per engagement while the process remains constant.

2. Methodology & Core Principles

The Valantic STI migration methodology is built on three foundational pillars that govern every decision, every line of translated code, and every architectural choice. These are not guidelines — they are non-negotiable engineering constraints enforced through automation.

2.1 The Five Iron Laws

These laws override all other considerations. Violation of any iron law is treated as a defect, regardless of schedule pressure or perceived simplicity of the code.

# Iron Law Rationale Enforcement
1 No translation without source analysis first Prevents shallow translations that miss control flow, edge cases, or implicit coupling Analysis artifacts required before plan generation is permitted
2 No module marked complete without semantic equivalence verification Ensures business logic preservation is proven, not assumed Verification gate with numerical scoring — must exceed threshold
3 No new dependency without checking if the platform already provides it Prevents dependency bloat and unnecessary third-party risk Migration Ladder enforced — must justify why higher rungs don't apply
4 No architectural decisions without formal record Prevents silent redesigns — every divergence from 1:1 translation is intentional and traceable Decision records are numbered, permanent, and part of the audit trail
5 No code without tests Behavioral equivalence requires proof; tests are that proof Automated hooks block file creation without corresponding test
Anti-Rationalization Principle

Common shortcuts are pre-emptively blocked: "This file is too simple to analyze" — NO, all files are analyzed. "I'll improve the architecture while migrating" — NO, migrate first, improve second. "This is obviously equivalent" — NO, verification is automated, not subjective.

2.2 The Migration Ladder

Before writing any target code, the engineer (or AI agent) must climb the ladder from bottom to top, stopping at the first rung that solves the problem. This ensures migrated code is as simple as possible and leverages platform capabilities rather than re-implementing them.

  1. Does this source code need to exist in the target?
    YAGNI — skip dead code, unreachable paths, deprecated features. Mark in inventory and move on.
  2. Does the target framework/platform already provide this?
    Use built-in capabilities. Don't rewrite what the platform gives for free.
  3. Does the target language's standard library cover it?
    Standard library types are stable, performant, and dependency-free.
  4. Does an already-chosen dependency handle it?
    Reuse what's already in the project. Don't add a second library for the same capability.
  5. Can a declarative mechanism do it?
    Annotations, attributes, decorators, configuration — declarative over imperative.
  6. Can it be one line?
    Write one line. No abstraction needed for one-liners.
  7. Only then: write the minimum implementation
    The simplest code that passes the test. Nothing more.
Adaptation Per Engagement

The ladder adapts to the output type. For libraries and SDKs, framework-dependent rungs are skipped — libraries must not depend on application frameworks at runtime. The ladder jumps from rung 1 directly to rung 3 (standard library).

2.3 Behavioral Equivalence Guarantee

The migrated code must produce identical observable behavior to the source. This is the primary correctness criterion — not code similarity, not structural equivalence, but behavioral equivalence.

Equivalence Requirements

Property Requirement
Input/Output Same inputs → same outputs (within type-system differences)
Error conditions Same error conditions → equivalent error handling (mapped, not swallowed)
Ordering guarantees If source was ordered, target must be ordered
Concurrency properties If source was thread-safe, target must be thread-safe
API contracts All public interfaces preserved or divergence explicitly recorded

Acceptable Divergences (Must Be Recorded)

Divergence Classification
Resource management differences (manual → platform-managed lifecycle) Expected
Platform-specific code replaced with portable equivalents Expected
Null/error safety improvements (weaker → stronger type safety) Improvement
Runtime performance characteristics (different memory model) Expected

Unacceptable Divergences (These Are Defects)

The following are treated as migration defects — they block module completion:

  • Silently dropping functionality
  • Changing business logic without a decision record
  • Swallowing errors that the source propagated
  • Breaking API contracts that other code depends on
  • Removing concurrency safety without analysis

3. Process Phases

The migration follows a strict phase-based workflow. Each module passes through the same sequence of activities, with automated quality gates between phases. The process is deterministic, resumable, and produces traceable artifacts at every step.

End-to-End Flow

Init → Detect → Map → Analyze → Plan → Execute → Verify → Review

Initialization (Init → Detect → Map) runs once per project. The module cycle (Analyze → Plan → Execute → Verify → Review) repeats for each module in the dependency-ordered roadmap. Session interruptions are handled by the Resume capability, which reads persistent state and re-enters at the correct point.

3.1 Initialization Phase

Purpose

One-time project kickoff: scan the source codebase, build knowledge graphs for architectural understanding, detect technologies, gather architecture decisions, generate a dependency-ordered roadmap, and create the target project skeleton.

Aspect Details
Duration 30 minutes – 2 hours (depending on codebase size)
Inputs Source code root path, target technology configuration
Activities
  1. Parallel source scanning (4 agents: headers/interfaces, implementations, build files, tests)
  2. Knowledge graph extraction (structural + semantic analysis)
  3. Community detection and coupling hotspot analysis
  4. Technology detection and dependency mapping
  5. Interactive architecture decisions (output type, database, API style, auth, deployment)
  6. Source → target structural mapping
  7. Dependency-ordered roadmap generation (modules named by business domain)
  8. Target project skeleton generation (compilable from first commit)
Outputs inventory.md, config.json, state.md, roadmap.md, decisions.md, mapping.md, knowledge graphs, research documents (technology stack, dependency map, risk matrix), compilable target project skeleton with architecture tests
Quality Gate All source files inventoried; graphs generated; skeleton compiles; architecture tests pass; roadmap covers all files

3.2 Analysis Phase (Per Module)

Purpose

Deep-analyze a specific module's source code from four parallel angles: data flow, design patterns, dependency graph, and risk assessment. Produces a comprehensive analysis consumed by the planning phase.

Aspect Details
Parallelism 4 specialized analyzer agents run simultaneously on the same source
Analysis Dimensions
AgentFocus
Data Flow TracerOwnership, lifetimes, data movement, shared state, resource management
Pattern DetectorDesign patterns, language idioms, error handling patterns
Dependency MapperCall graph, interface surface, circular dependencies, external coupling
Risk AssessorUnsafe operations, platform-specific code, concurrency hazards, migration complexity
Outputs Analysis document — 4 sections + synthesis (complexity score, top challenges, recommended strategy, effort estimate)
Quality Gate All source files read; all 4 sections populated; every type/class documented; risks flagged with severity

3.3 Planning Phase (Per Module)

Purpose

Generate executable, dependency-ordered translation plans organized into waves that follow the target architecture's layer structure. Each plan is a self-contained work unit.

Aspect Details
Wave Strategy Waves follow target architecture dependency direction (inside-out):
domain/model → domain/port → domain/service → application → adapters-out → adapters-in → configuration
Plan Contents Metadata (module, wave, source/target files, complexity, required tests) + body (objective, source analysis, translation table, dependency mapping, target structure, test strategy, verification criteria)
Strategies Conservative: maximize 1:1 mapping
Modern: leverage modern target language features aggressively
Hybrid (default): conservative for business logic, modern for infrastructure
Quality Gate Every source file mapped to at least one plan; wave ordering respects dependencies; human engineer confirms plan before execution

3.4 Execution Phase (Per Module)

Purpose

Execute translation plans using wave-based parallel agents. Each wave completes and passes a build/test gate before the next wave begins. Failed gates trigger auto-fix or stop execution.

Aspect Details
Parallelism Plans within the same wave execute in parallel (no inter-dependencies within a wave)
Per-Wave Gate BLOCKING — all must pass before next wave starts:
  • Full compilation (zero errors)
  • All tests pass
  • Architecture rules pass (layer enforcement)
  • Coverage threshold met (if configured)
Failure Handling Spawn fixer agent (max 2 attempts per failure). If still failing → STOP, report to human.
Commit Strategy One atomic commit per plan with standardized message format for traceability
Quality Gate All plans executed; every file compiles; every test passes; architecture tests pass; domain layer has zero framework imports

3.5 Verification Phase (Per Module)

Purpose

Multi-dimensional verification gate: semantic equivalence (does the target do what the source did?), architecture quality (does it meet standards?), and graph-based drift detection (has the architecture shifted from the source?).

Verification Stage What It Checks Pass Threshold
1. Semantic Equivalence Method-level comparison, behavioral paths, edge cases, API contract preservation ≥85% (fail) / ≥95% (clean pass)
2. Architecture Quality Layer compliance, target platform standards, code quality metrics, test quality, minimalism audit ≥80% (fail) / ≥90% (clean pass)
3. Graph-Based Verification Source↔Target coverage, structural equivalence, coupling drift, layer validation ≤25% drift (warn) / ≤10% (clean pass)
Verification Outcome

PASS: Advance to review → PASS_WITH_NOTES: Show advisory to engineer → FAIL: List blocking issues, do NOT advance. All findings recorded in verification report.

3.6 Review Phase (Per Module)

Purpose

Final quality gate with two specialized review passes. After approval, the module is tagged complete in version control. On the final module, ARC42 architecture documentation is generated automatically.

Pass Focus Verdicts
Pass 1: Semantic Fidelity Undocumented behavior changes, dropped functions, error path differences, concurrency preservation FAITHFUL / DRIFT_MINOR / DRIFT_MAJOR
Pass 2: Minimalism Audit Dead code, re-implemented platform functionality, unnecessary abstractions, over-engineering APPROVED / OVER_ENGINEERED / UNNECESSARY / SIMPLIFIABLE

Resolution Paths

Verdict Action
APPROVED Tag in VCS, advance to next module
NEEDS_FIXES Auto-fix or present to engineer, re-verify, re-review
BLOCKED DRIFT_MAJOR items require explicit resolution or decision record
Final Module Completion

When the last module passes review, the system automatically generates full ARC42 architecture documentation (12 sections), runs a final knowledge graph comparison with source↔target analysis, and commits the documentation to the target project.

4. Architecture & Technical Standards

The framework enforces an opinionated target architecture from the first line of generated code. Architecture is not a recommendation — it is enforced by automated architecture tests that run on every compilation. Violations are build failures, not review comments.

4.1 Target Architecture (Ports & Adapters / Hexagonal)

All service-type migrations land in a layered architecture with strict dependency direction. The domain layer is pure business logic with zero framework imports. Dependencies always point inward. This is enforced automatically.

Layer Contains Allowed Dependencies Forbidden
Domain Entities, value objects, domain services, port interfaces Language standard library only Any framework, any external library
Application Use cases (orchestration), DTOs, mappers Domain layer, transaction management Adapter layer, persistence entities
Adapter IN (Driving) API controllers, CLI commands, message listeners, schedulers Domain ports (in), Application DTOs Domain services (implementations), Adapter OUT
Adapter OUT (Driven) Repository implementations, HTTP clients, message publishers, file I/O Domain ports (out), persistence frameworks, external clients Adapter IN, Application use cases directly
Enforcement Mechanism

Architecture tests are generated during project initialization and run on every build. A domain class importing framework packages fails the build. An adapter accessing another adapter fails the build. This is not discoverable in code review — it is a hard compilation gate.

4.2 Output Type Profiles

Not every legacy codebase becomes a web service. The framework recognizes four distinct output types and adapts architecture, dependencies, testing strategy, and build configuration accordingly.

Output Type Source Was Target Becomes Key Characteristics
Service Application with network/IPC listeners Modern web service (deployable) Hexagonal architecture, observability, health endpoints, API contracts
Library Shared library / static lib / DLL Packaged library (no main, no framework) api/internal/spi layering, module boundaries, zero framework lock-in
SDK Library with versioned public API Library + documentation + stability contracts + samples Stability annotations, binary compatibility checking, 100% documented API
CLI Command-line tool Modern CLI with native compilation option stdout/stderr contracts, exit codes, fast startup, shell completions

4.3 Standards Adaptation Per Engagement

The methodology separates universal principles (which never change) from technology-specific standards (which are configured per engagement). This allows the same process to apply regardless of source/target technology.

Universal (Always Applies) Configured Per Engagement
Five Iron Laws Source language and version
Migration Ladder logic Target language, version, and framework
Behavioral equivalence verification Target architecture pattern (hexagonal, clean, vertical slices)
Wave-based execution (inside-out) Specific architecture test rules
10 quality gates per module Test framework and coverage tools
Decision records Build tool and dependency management
Knowledge graph analysis Client naming conventions and domain terminology
ARC42 documentation generation Deployment target and observability stack
Two-pass review (fidelity + minimalism) Security model and auth patterns
Audit trail (.migration/ directory) Client-specific code quality thresholds

4.4 Code Quality Constraints

Regardless of target language, universal quality constraints are enforced. Specific thresholds may be adjusted per client, but the categories are always measured.

Metric Default Limit Enforcement

File/class length ≤200 LOC Review gate
Function/method length ≤30 LOC Review gate
Dependency count (per component) ≤4 injected dependencies Review gate (signals component does too much)
Test coverage (business logic) 95% line / 90% branch CI gate

5. Quality Assurance & Automation

Quality is not a phase — it is enforced continuously through automated hooks, compilation gates, architecture tests, and multi-dimensional verification. Human review is the last line of defense, not the first.

5.1 Automated Quality Hooks

Hooks run automatically during development, catching violations at the moment of introduction rather than during review.

Hook Trigger Action
Compile-on-save Any target source file saved Runs incremental compilation immediately
Domain purity check File created in domain layer Rejects framework imports — zero external dependencies in domain
Architecture direction check File saved in adapter layer Validates dependency direction — adapters may only invoke port interfaces
Test companion reminder Any new source file created Verifies corresponding test file exists

5.2 Quality Gates (10 Per Module)

Each module passes through 10 quality gates before it is marked complete. Gates are blocking — a failed gate stops progression until resolved.

# Gate When Threshold
1 Compilation After every wave Zero errors
2 Unit tests pass After every wave 100% green
3 Architecture tests (layer compliance) After every wave Zero violations
4 Coverage threshold After execution Configurable (default: 95%/90%)
5 Semantic equivalence score Verification ≥85% (pass) / ≥95% (clean)
6 Architecture quality score Verification ≥80% (pass) / ≥90% (clean)
7 Graph coverage Verification ≥75% source concepts covered
8 Architecture drift Verification ≤25% (warn) / ≤10% (clean)
9 No DRIFT_MAJOR unresolved Review Zero remaining
10 Minimalism audit passed Review No OVER_ENGINEERED findings

5.3 Architecture Tests — Architecture as Code

Architecture tests are generated during project initialization and provide compile-time enforcement of structural rules. They are output-type-aware and adapt per engagement.

Service Profile — Example Rules
  1. Domain classes must not import framework packages
  2. Domain classes must not import persistence packages
  3. Ports must be interfaces (not classes)
  4. Domain layer depends on nothing outside domain
  5. Application layer must not import adapter packages
  6. Driving adapters only use port-in interfaces
  7. No field injection (constructor injection only)
  8. Use cases reside in application layer
Library/SDK Profile — Example Rules
  1. Domain purity (no framework imports)
  2. Public API has no framework annotations
  3. Public API does not depend on internal packages
  4. No application framework bootstrap class
  5. No web/API controller classes
  6. Internal packages not exported through module system
  7. Extension point interfaces have default implementations
CLI Profile — Example Rules
  1. Domain must not import framework packages
  2. Domain must not import CLI framework packages
  3. Dependencies point inward
  4. CLI adapter only uses port-in and DTOs
  5. No web controllers
  6. No application framework bootstrap

5.4 Test-First Migration

Every translated file ships with tests that exercise the same paths as the source original. The test porting follows systematic mappings — every source test pattern has a documented target equivalent configured per engagement.

Test Porting Principle

Source test structures are mapped systematically to target equivalents: fixtures → setup methods, parameterized tests → data-driven tests, mock expectations → mock verification, sections/contexts → nested test groups. The mapping is documented per engagement and applied consistently across all modules.

5.5 Test Pyramid

Level Scope Speed
Unit Domain + Application layer (pure logic, mocked dependencies) Milliseconds
Integration (Slice) Individual adapters with real infrastructure (containers/mocks) Seconds
End-to-End Full application context 10-30 seconds
Architecture Structural rules (layer enforcement) Milliseconds
Contract API compatibility with consumers Seconds
Behavioral (BDD) Business scenarios in client's language Seconds

6. Tooling & AI Agent Strategy

AI is used as a force multiplier under strict governance, not as an autonomous decision-maker. The methodology deploys specialized AI agents for mechanical tasks (scanning, translation, verification) while keeping architectural decisions, risk assessment, and approval authority with human engineers.

Design Philosophy

AI agents perform the volume work (reading thousands of files, translating code, checking equivalence) while humans perform the judgment work (architecture decisions, risk acceptance, plan approval). The framework constrains AI behavior through iron laws, quality gates, and strict output format requirements — AI cannot bypass these constraints.

6.1 Multi-Agent Architecture

Rather than using a single AI agent for everything, the framework deploys specialized agents matched to each task's complexity and compute requirements.

Agent Role Model Tier Parallelism Purpose Constraints
Scanner Fast/cheap ×4 parallel File classification and inventory Read-only on source, write only to state directory
Detector Standard ×1 Technology and dependency detection Must classify every dependency, provide target equivalent
Analyzer Most-capable ×4 parallel Deep code analysis (data flow, patterns, deps, risks) Must document every type/class, flag all risks with severity
Translator Most-capable Per-wave parallel Code translation following plans Iron Laws enforced: read ALL source first, follow plan exactly, domain purity, every public function tested
Verifier Most-capable ×1 Semantic equivalence checking Verdicts: EQUIVALENT / MINOR_DIFF / SEMANTIC_DRIFT
Reviewer Most-capable ×2 passes Fidelity review + minimalism audit Two separate passes with different objectives
Fixer Standard ×1 Targeted repair of gate failures Max 2 attempts, cannot redesign, cannot change public API

6.2 Agent-Agnostic Framework

The methodology is not locked to any single AI coding tool. The same standards, workflows, and quality hooks are delivered through thin adapters to whichever AI platform the engagement uses. This ensures no vendor lock-in at the tooling level.

One Source, Multiple Outputs

The knowledge base (standards, workflows, hooks) is maintained once in portable markdown. Agent-specific installers format this content for each platform's native conventions. Methodology updates propagate to all platforms simultaneously. Currently supported: Kiro, Claude Code, Codex — extensible to any agent with file-based configuration.

6.3 Knowledge Graph Tooling

Two complementary knowledge graph tools provide deep codebase understanding. They run automatically at defined points in the workflow — no manual invocation required.

Tool Role Unique Capabilities Integration Points
Graphify Analytical backbone Community detection (Leiden algorithm), coupling hotspot analysis, edge confidence tiers, hidden dependency discovery, graph traversal queries Init (source graph), Execute (target graph), Verify (comparison), Review (ARC42)
Understand-Anything Human interface Interactive dashboard, guided tours (dependency-ordered), business domain extraction, onboarding guide generation, diff impact analysis Init (source understanding), on-demand throughout

How They Complement Each Other

  • Graphify communities → inform module grouping (dependency-based boundaries)
  • UA domain flows → inform module naming (business-meaningful, not technical)
  • Graphify coupling hotspots → quantify migration risk per component
  • UA layer classification → auto-assign source files to target architecture layers
  • Both use AST extraction — structural analysis without LLM token cost
  • Cross-referencing — disagreements between tools signal architectural concerns

Graph-Based Migration Tracking

Mode Purpose Metrics Produced
source Baseline architecture graph of legacy codebase Communities, coupling hotspots, module boundaries
target Evolving architecture graph of modern target (incremental) New communities, layer validation, architecture compliance
compare Source↔Target structural comparison Coverage %, drift %, lost/new connections, complexity delta
arc42 Generate architecture documentation from graph data 12 ARC42 sections with diagrams, auto-populated

7. Risk Management & Mitigation

Risk is not an afterthought — it is identified, quantified, and mitigated at every stage. The methodology provides three interlocking risk management mechanisms: upfront risk assessment, continuous decision tracking, and automated drift detection.

7.1 Risk Identification & Scoring

During initialization, the technology detection phase produces a formal risk matrix scoring every identified risk using impact × likelihood.

Risk Category Examples Detection Method Mitigation Approach
Platform-specific code OS-specific APIs, hardware interfaces, conditional compilation Pattern scanning, platform API detection Portable target equivalents identified; decision recorded
Unsafe operations Manual memory management, unsafe casts, raw resource handles Risk assessor agent (pattern matching) Type-safe target replacements; lifecycle analysis ensures no leaks
Concurrency hazards Shared mutable state, lock-free algorithms, custom synchronization Data flow analysis + concurrency pattern detection Modern concurrency primitives; safety properties preserved
No target equivalent Platform-specific libraries, hardware drivers, assembly code Dependency mapping with confidence scoring Decision required: FFI bridge, alternative library, or feature removal (with record)
Coupling hotspots Components referenced by >30% of codebase, circular dependencies Knowledge graph community detection + coupling analysis Decomposed during planning; interface extraction; prioritized in early waves
Metaprogramming / code generation Macros, templates, reflection-heavy code, compile-time generation Pattern detection in analysis phase Manual review + explicit expansion; annotated for human attention

7.2 Decision Records (Architectural Traceability)

Every architectural divergence from 1:1 translation is recorded as a numbered, permanent decision. This provides complete traceability for audits and ensures no silent redesigns.

Decision Record Format
## D-007: Replace async I/O library with target platform standard

**Context:** Source uses third-party async I/O library. Target platform provides
equivalent async capabilities natively, making the dependency unnecessary.

**Decision:** Use target platform's standard async I/O instead of adding equivalent
third-party library.

**Consequences:** Simpler dependency graph. Platform-native concurrency model.
No custom reactor pattern needed.

**Migration Ladder Rung:** 3 (standard library covers it)

Rules governing decisions:

  • Numbered sequentially (D-001, D-002, …) — never reordered or deleted
  • Must reference the Migration Ladder rung that justifies the choice
  • Must state consequences (what changes for downstream code)
  • Created by any agent that encounters a non-obvious translation choice
  • Reviewed by human engineer during plan approval

7.3 Automated Drift Detection

Architecture drift is detected quantitatively through knowledge graph comparison, not through subjective assessment.

Drift Metric What It Measures Threshold
Coverage % % of source concepts that have a target equivalent ≥75% (per module)
Structural preservation Community count stability (±20% tolerance) No dramatic changes without justification
Coupling drift New coupling hotspots in target that didn't exist in source Zero new hotspots (warning if any appear)
Lost connections Relationships in source with no target equivalent Must be justified (dead code, or decision record)
Layer violations Domain→adapter paths in target graph Zero (hard failure)

7.4 Failure Handling & Recovery

Failure Scenario Automatic Response Escalation
Compilation failure after wave Fixer agent (max 2 attempts) STOP execution, report to engineer
Test failure Fixer analyzes and attempts targeted repair After 2 fails → BLOCKED, human intervention
Semantic drift detected DRIFT_MAJOR finding with location Must be resolved or accepted via decision record
Session interruption State persisted; stall detection after 10 min Resume re-enters at correct point with full context
Uncertain translation Agent marks for human review Flagged in review pass for decision
Escalation Principle

The fixer agent has strict boundaries: max 2 attempts, cannot redesign, cannot change public API, cannot weaken tests. If automated repair fails, the system stops and reports rather than producing dubious code. We prefer blocking progress over producing incorrect output.
